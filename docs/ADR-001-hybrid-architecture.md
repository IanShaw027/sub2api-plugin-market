# ADR-001：插件市场架构边界（Hybrid）

- 状态：Accepted
- 日期：2026-03-04
- 负责人：Platform Architecture
- 关联仓库：`sub2api`、`sub2api-plugin-market`

## 1. 背景

`sub2api` 需要持续瘦身并保持网关核心稳定；插件市场需要独立演进、独立部署、独立治理。
当前存在“内嵌控制平面骨架”与“独立插件市场服务”并行的双轨状态，容易造成：

1. API 契约分叉
2. 文档与实现不一致
3. 责任边界不清
4. 交付与运维成本上升

## 2. 决策

采用 **Hybrid 架构**：

1. `sub2api-plugin-market` 作为**唯一控制平面**
2. `sub2api` 作为**数据平面 + 运行时执行平面**

### 2.1 控制平面职责（仅 `sub2api-plugin-market`）

1. 插件元数据管理（列表、详情、版本）
2. 工件分发（预签名 URL）
3. 审核流与发布状态
4. 信任密钥分发与轮换元数据
5. 市场侧审计与统计

### 2.2 数据平面职责（仅 `sub2api`）

1. 插件安装状态（desired/effective）
2. 本地 lockfile 与重启生效流程
3. 运行时加载、调度、隔离、熔断
4. 运行时审计与观测
5. 安装前验签、兼容性校验

## 3. 明确排除项

1. 不再在 `sub2api` 中演进“完整市场控制平面”功能
2. 不在第一阶段引入运行中热更新（统一重启生效）
3. 不允许未签名或不受信插件进入安装链路

## 4. 契约与版本治理

1. 市场 API 使用语义化版本：`/api/v1/...`
2. `plugin_api_version` 与宿主能力窗口独立治理
3. 错误码建立注册表，禁止跨仓私自复用同码不同义
4. 破坏性变更必须先出 RFC/ADR，再进实现

## 5. 实施要求（P0）

1. 两仓文档统一到单一事实源（路径、字段、错误码）
2. `sub2api-plugin-market` CI 必须保证 `go test ./...` 通过
3. 管理后台审核接口前后端字段保持双向兼容
4. 发布模式下禁止弱默认 JWT 密钥

## 6. 验收标准

1. 两仓不存在相互冲突的“控制平面主实现”声明
2. `sub2api-plugin-market` 测试与 CI 绿灯
3. 文档抽样命令可按文档执行得到预期行为
4. 关键接口（审核、下载、密钥）契约一致

## 7. 回滚策略

1. 通过 feature flag 关闭新接入路径，保留核心代理能力
2. 控制平面变更失败时，网关保持“无插件市场依赖”可运行
3. API 层维持旧字段兼容窗口，避免控制台中断
